# HausmanHub 0.5.8: ограничения входных данных для планшета

Дата: 2026-07-18.

## Что изменено

Публичный home-контракт HausmanHub получил версию 3. В `control` каждой комнаты рядом
со списком `actions` добавлено поле `action_inputs`. Для
`set_room_target.target_temperature` оно сообщает:

- тип `number`;
- обязательность значения;
- минимум 18 °C;
- максимум 28 °C;
- шаг 0,5 °C;
- единицу `°C`.

Выключение комнаты не требует входного значения. Если текущий Climate API не
объявляет команду изменения температуры, HausmanHub не объявляет и её входные данные.

## Зачем это нужно

Android сможет построить правильный регулятор температуры из ответа HausmanHub. Ему
не придётся отдельно зашивать диапазон climate-core и предлагать значение,
которое сервер затем отклонит.

Проверка команды и публичное описание используют одни константы. Финальной
границей всё равно остаётся серверная проверка каждого запроса.

## Совместимость и безопасность

Строгая JSON Schema и синтетический пример находятся в `contracts/v3` и
`fixtures/hausmanhub_climate_v3`. Прежние home-схемы v1 и v2 остаются в HACS-пакете,
чтобы старые формы ответа распознавались явно.

Изменение не включает климатический мост, не создаёт реестр живого дома, не
меняет Android-репозиторий и не отправляет физические команды. Развёртывание
должно оставить мост в `disabled`.

## Проверка

Подготовленный пакет прошёл 232 местных теста, полную проверку HACS-пакета и
границ файлов. Disposable Home Assistant Core 2026.6.4 и 2026.7.0 прошли полный
сценарий с настоящей локальной аутентификацией, home v3 и измеренным нулём
климатических command POST.

Kimi model `kimi-for-coding/k2p7` проверил точный staged diff только на чтение
в сессии `ses_08b312059ffedrMEVGxBLevcNI`: PASS, существенных замечаний нет.
Два наблюдения признаны стилистическими и неблокирующими: JSON не различает
`18` и `18.0`, а пустой `action_inputs` намеренно остаётся явной частью
строгого контракта. Перед публикацией остаётся GitHub Actions.
