# HausmanHub 1.9.8: собственное применение желаемого состояния

Дата: 2026-07-21. Пункт плана: 36, подшаг 36d.

## Что сделано

Применение настроек климата больше не использует внешний Climate API.
Ручное применение контура, переключение дневного и ночного профиля по
расписанию, временная температура и возврат к расписанию работают по
собственной цепочке: сохранить желаемое состояние контура, построить
собственное наблюдение Home Assistant, вычислить нативный план, проверить
всю целевую область, выполнить строгие вызовы Home Assistant через единый
исполнитель и коротко (до двух секунд) сверить результат наблюдением.
POST `/endpoint/climate/api/v1/command` во внешний модуль для этих
операций больше не выполняется; его сохраняет только старый канареечный
маршрут `/actions`.

Новый чистый планировщик `application/climate_application.py` (модели в
`climate_application_models.py`) получает сохранённый контур, реестр,
метку режима, собственный снимок наблюдения, отпечаток желаемого
состояния и целевые комнаты, а возвращает изоляцию, сравнение, план
строгих вызовов, покомнатный результат защит, плоский детерминированный
ряд вызовов, изначально согласованные комнаты и ограниченные причины
отказа. Home Assistant он не импортирует и исполнительной власти не имеет.

Граница исполнителя обобщена: `ClimateTrialCallExecutor` стал
`ClimateStrictHaCallExecutor`; `HomeAssistantClimateCallExecutor`
задокументирован как единый строгий исполнитель климатических вызовов
для пробной комнаты, минутного управляемого контура и применения
настроек. Скелетная проверка по-прежнему находит `hass.services.async_call()`
только в `switch.py` и `climate_ha_executor.py`.

## Правила перехода

- Ручное применение и расписание атомарны по всему контуру: одна
  неготовая, неуправляемая, неготовая по наблюдению или неполностью
  переведённая комната отменяет все вызовы разом. Временная температура
  проверяет только свою комнату.
- Режимы `disabled`, `shadow` и `canary` отклоняют операции до чтения
  состояний. Работает только режим `managed`.
- Порядок сохранения: ручное применение не пишет контур повторно;
  расписание сохраняет профили и маркер периода до планирования;
  временная температура сохраняет (или снимает) переопределение до
  планирования. Отклонённое расписание не повторяется таймером.
- После исполнения наблюдение опрашивается сразу и до десяти раз по
  200 мс без повторных вызовов. Повторный `request_id` только перечитывает
  наблюдение и может повысить квитанцию до `confirmed`.
- Квитанция `hausman-hub-climate-control-receipt` версии 1 не изменилась:
  `confirmed`, `pending`, `partial`, `unavailable` с причинами
  `already_in_sync`, `state_not_confirmed`, `verification_unavailable`,
  `command_result_unavailable`, `engine_rejected`. Локальные дельты
  (`changes.temperature`, `changes.strategy`) считаются по сохранённому
  желаемому состоянию; `changes.automatic_mode` всегда ноль.
- Минутный управляемый контур остаётся независимым и может позже
  повторить расходящийся план; это осознанное решение.

## Что не изменилось

Внешний Climate API обслуживает публичный снимок Android, предпросмотр
применения, готовность, мастера настройки, теневые доказательства и
предполётные проверки канарейки. Их перевод: подшаги 36e-36g. Работающий
климатический модуль и Android-приложение не изменялись.

## Проверки

- 571 локальный тест, включая: применение всего контура по принципу
  «всё или ничего», временная температура при сломанной соседней комнате,
  режимные отказы до чтения состояний, работа с `bridge_client=None`,
  «ядовитый» мост без единого вызова во всех четырёх операциях, порядок
  сохранения, отказ таймера повторять отклонённое расписание, отображение
  нулевого и частичного исполнения, подтверждение в пределах опроса,
  таймаут без повтора, сломанное наблюдение после вызовов, дубликаты
  только с перечитыванием, неизменная форма и редактирование квитанции,
  административные сохранения без команд.
- `tools/check_local_release.py`: результат фиксируется в финальном отчёте.
- Одноразовые Home Assistant Core 2026.6.4 и 2026.7.0: результат
  фиксируется в финальном отчёте.
- Независимый просмотр: вердикт фиксируется после просмотра.
