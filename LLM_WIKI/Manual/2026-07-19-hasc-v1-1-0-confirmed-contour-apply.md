# HASC 1.1.0: подтверждаемое применение настроек контура

Дата: 2026-07-19.

## Решение

HASC начинает управлять климатом через существующий `hausman-climate`, не
перенося и не дублируя его алгоритм. Обычный пользователь сохраняет
автоматический климатический контур, затем отдельным действием просматривает и
подтверждает применение поддерживаемых параметров.

Для обычного контура введён внутренний режим связи `managed`. Он нужен, чтобы
не нарушать прежние гарантии: `shadow` по-прежнему не отправляет POST, а
`canary` остаётся отдельным техническим управлением одной комнатой.

## Разрешённое применение

Из сохранённого контура HASC строит максимум три закрытые типизированные команды
на комнату:

1. `set_room_target_strategy`;
2. `set_room_target`;
3. `set_room_mode auto`.

Автоматический режим идёт последним. Пользователь, Android и HTTP-запрос не
могут задать backend command type, source ID, entity ID, Home Assistant service,
адрес или произвольный payload. Старый общий actions-route в режиме `managed`
закрыт; применять можно только целиком сохранённые параметры контура.

Цель влажности не отправляется. Climate API v1 сообщает влажность, но его
общая команда влажности комнаты не является поддерживаемой границей HASC 1.1.
Preview и публичный статус явно сообщают это ограничение.

## Квитанция и подтверждение

Запрос содержит только ограниченный `request_id`, `contour_id=climate` и
`confirm=true`. До первого POST HASC резервирует непрозрачный `operation_id`.
Одинаковый повтор возвращает существующую квитанцию; конфликтующее повторное
использование request ID отклоняется. Потерянный или неоднозначный ответ не
вызывает повторной отправки.

Команды выполняются последовательно. Явный отказ останавливает применение.
Если часть команд уже принята, результат остаётся `partial`. После всех
принятых команд HASC снова читает Climate API и подтверждает комнату только
при видимом совпадении температуры, стратегии и авто-режима. Иначе квитанция
остаётся `pending`; повтор с тем же request ID может только перечитать состояние.

## Публичная граница

- `GET /api/hausman_hub/v1/contours` теперь отдаёт контракт v2 с фактической
  стратегией и `execution.settings_apply`.
- `GET /api/hausman_hub/v1/contours/apply-preview` показывает только количество
  комнат и поддерживаемых изменений.
- `POST /api/hausman_hub/v1/contours/apply` возвращает версионную квитанцию без
  приватных идентификаторов и backend-содержимого.

Маршруты доступны только точной локальной роли планшета Home Assistant,
требуют аутентификацию, запрещают CORS и кэширование.

## Граница публикации

Тесты используют только синтетический Climate API. Установка версии через HACS
не создаёт контур, не переводит старую настройку в `managed`, не вызывает apply
и не отправляет живые климатические команды. Для старого контура 1.0 владелец
один раз повторно сохраняет мастер, а реальное применение всё равно требует
отдельного подтверждения.

Следующий этап — индивидуальные параметры комнат, расписания и профили, затем
экран Android поверх уже версионных contour/apply контрактов.
