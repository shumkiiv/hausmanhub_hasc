# Климатический контур HASC 1.x

Версия документа: 2026-07-19. Реализация: HASC 1.4.0.

## Простая схема

```text
Android-планшет
       ↕ локальный API HASC
HASC: комнаты, выбранные устройства, цели, расписание и понятный статус
       ↕ фиксированный Climate API v1
существующий hausman-climate: алгоритм, защита и выполнение
       ↕
Home Assistant / Node-RED / физические устройства
```

Пользователь работает с HASC, а не собирает автоматику вручную. Он выбирает
климатический контур, комнаты и устройства, задаёт параметры и затем видит
единое состояние. Существующий климатический контур остаётся двигателем под
HASC: он уже умеет работать с кондиционерами, термоголовками и увлажнителями,
учитывает авто- и ручные режимы, ночные профили, задержки, ограничения,
доступность и фактическую обратную связь.

HASC не дублирует этот зрелый алгоритм. Линия 1.x постепенно оборачивает его
универсальным интерфейсом, а Android знает только HASC.

## Как настроить

1. В Home Assistant откройте **Настройки → Устройства и службы → HASC →
   Настроить**.
2. Выберите **Климат**.
3. Выберите **Выбрать комнаты и климатические устройства**.
4. Если HASC ещё не знает систему климата, введите её локальный адрес вида
   `http://192.168.1.10:1880`. Разрешён только буквальный локальный IP без
   пути, параметров, логина и пароля.
5. Выберите комнаты. Затем выберите находящиеся в них устройства. HASC
   показывает формат **«Комната — Устройство»** и не показывает служебные ID.
6. Для каждой выбранной комнаты HASC откроет отдельный короткий экран. Задайте:

   - температуру 18–28 °C, шаг 0,5 °C;
   - влажность 30–70 %, шаг 5 %;
   - мягкую, обычную или быструю стратегию.

   Название контура и режим работы выбираются один раз на первом экране.

7. Проверьте резюме: названия комнат, их отдельные цели, число устройств,
   состояние двигателя и причины, которые требуют внимания.
8. Подтвердите сохранение.
9. Вернитесь в **Климат** и выберите **Задать настройки для дня и ночи**.
   Для каждой комнаты задайте оба набора. Сохранение профилей
   не отправляет команды.
10. Выберите **Настроить автоматическое расписание**, задайте начало дня и ночи.
11. Подтвердите постоянное разрешение. После этого HASC сам выберет нужные
    настройки по местному времени Home Assistant и передаст их подключённой
    системе климата.

При повторном редактировании HASC заранее отмечает сохранённые комнаты и
устройства. Для каждой комнаты он подставляет сохранённые настройки HASC, а не
временно отличающееся значение двигателя. Редактирование обычных параметров
меняет активный профиль и сохраняет второй без изменений.

В 1.4.0 профиль переключается автоматически сразу для всех комнат. Проверка
времени выполняется раз в минуту, но команды отправляются только при настоящей
смене **«День»/«Ночь»**. При включённом расписании ручной выбор закрыт; сами
дневные и ночные значения можно редактировать.

## Что означают режимы

- **Наблюдение** — HASC показывает состояние выбранного контура и сравнивает
  цели, но не считает его автоматически активным со стороны HASC.
- **Автоматически** — сохранённые температуру, стратегию и авто-режим можно
  отдельно применить к существующему двигателю. Подтверждение работы появляется
  только при свежих данных, доступных устройствах, готовом праве управления и
  фактически совпадающих значениях.
- **Выключен в HASC** — описание и назначения сохраняются, но HASC не показывает
  контур как активный. Внешний двигатель при этом не выключается.

Если все найденные комнаты уже работают в автоматическом режиме, мастер
предлагает режим **Автоматически** по умолчанию. Иначе безопасным значением
будет **Наблюдение**.

## Что HASC хранит

HASC хранит две связанные части:

- приватный реестр комнат и устройств;
- публичное описание контура и его параметров.

Они проверяются и сохраняются вместе. Если запись одной части не удалась,
предыдущая рабочая настройка восстанавливается.

Устройства имеют понятный тип: кондиционер, радиаторная термоголовка,
увлажнитель, тёплый пол, датчик температуры или датчик влажности. Для устройств,
которыми уже владеет `hausman-climate`, HASC не просит повторно выбирать Home
Assistant entity: приватная связь остаётся у существующего двигателя.

## Статус контура

Локальный маршрут `GET /api/hausman_hub/v1/contours` возвращает контракт
`hausman-hasc-contours` версии 4. Для каждого климата доступны:

- имя, тип, режим и общее состояние;
- комнаты, текущая температура и влажность;
- активные цели и стратегия;
- выбранный профиль и оба набора **«День»/«Ночь»**;
- признак расписания и время начала дня и ночи;
- количество назначенных устройств;
- режим существующего двигателя;
- фактическая стратегия существующего двигателя;
- готовность его права управления;
- совпадение целей;
- подтверждение автоматической работы;
- доступность явного применения поддерживаемых настроек;
- ограниченный список понятных причин.

В ответе нет адреса двигателя, `source_id`, `entity_id`, команд нижнего уровня
или приватного содержимого Home Assistant. Маршрут требует аутентификацию Home
Assistant, локальный источник, запрещает CORS и не кэшируется.

## Как HASC 1.4 применяет настройки

Обычный автоматический контур использует отдельный внутренний режим связи
`managed`. Он не меняет смысл прежнего `shadow`: режим **«проверка без команд»**
по-прежнему никогда не отправляет POST. Технический `canary` также остаётся
отдельным старым инструментом одной комнаты.

Перед применением HASC берёт значения выбранного профиля, читает свежее
состояние и строит только необходимые команды. Порядок фиксирован для каждой
комнаты:

1. `set_room_target_strategy` — мягкая, обычная или быстрая стратегия;
2. `set_room_target` — сохранённая температура;
3. `set_room_mode auto` — автоматический режим включается последним.

Команды строятся из сохранённого контура. Планшет и форма Home Assistant не
могут передать тип backend-команды, приватный идентификатор, entity, сервис или
произвольный payload.

До первой отправки HASC резервирует `request_id` и непрозрачный `operation_id`.
Повтор одинакового запроса возвращает прежнюю квитанцию без нового POST.
Неоднозначный сетевой результат также не переотправляется. После всех принятых
команд HASC перечитывает состояние и считает комнату подтверждённой только при
совпадении температуры, стратегии и автоматического режима.

Цель влажности не отправляется: существующий Climate API пока не имеет общей
команды влажности комнаты. Это ограничение присутствует в preview-контракте.

Для Android доступны два локальных маршрута с той же строгой ролью планшета:

- `GET /api/hausman_hub/v1/contours/apply-preview`;
- `POST /api/hausman_hub/v1/contours/apply` с точными полями `request_id`,
  `contour_id=climate` и `confirm=true`.

Установка или обновление HASC эти маршруты не вызывает и живые настройки не
применяет. Автоматическое расписание использует тот же исполнитель, но только
после отдельного постоянного подтверждения владельца. Сначала HASC сохраняет
новый выбранный профиль, затем выполняет одну попытку применения. Неопределённый
ответ не приводит к автоматической повторной отправке.

## Следующие этапы 1.x

1. Добавить временное ручное отклонение до следующего переключения расписания.
2. Подключить экран Android к контракту климата: состояние, объяснение,
   изменение параметров и временный режим.
3. Добавить непрерывный диспетчер контура, журнал понятных решений и разрешение
   конфликтов между несколькими типами контуров.
4. Добавить следующие контуры на общей основе без изменения принципа работы
   приложения.

## Дополнительные настройки

Старые экраны реестра, bridge/shadow/canary, встроенного предпросмотра 0.6 и
служебного переключателя сохранены для миграции, диагностики и разработки.
Они находятся в **Диагностике и обслуживании** и не нужны для обычной настройки
климата.

Пробное физическое управление одной комнатой остаётся отдельным техническим
инструментом старого rollout. Публикация или установка 1.4.0 сама не включает
его и не применяет настройки автоматического контура.
