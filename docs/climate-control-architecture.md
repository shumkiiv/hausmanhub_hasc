# Климатический контур HausmanHub 1.x

Версия документа: 2026-07-20. Реализация: HausmanHub 1.7.7 в разработке.

## Простая схема

```text
Android-планшет
       ↕ локальный API HausmanHub
HausmanHub: комнаты, выбранные устройства, цели, расписание и понятный статус
       ↕ временный фиксированный Climate API v1 для проверки совпадения
существующий hausman-climate: эталон поведения только на время переноса
       ↕
Home Assistant / Node-RED / физические устройства
```

Пользователь работает с HausmanHub, а не собирает автоматику вручную. Он выбирает
климатический контур, комнаты и устройства, задаёт параметры и затем видит
единое состояние. Сейчас существующий климатический контур временно остаётся
двигателем под HausmanHub и служит неизменяемым эталоном уже работающего поведения.

Следующий этап — перенести его алгоритм, защиты и выполнение внутрь HausmanHub,
сверить решения двух реализаций без двойных команд, затем убрать обязательный
внешний Climate API. Итоговая установка должна состоять из HausmanHub и выбранных в
нём устройств; Android всегда знает только HausmanHub.

## Как настроить

1. В Home Assistant откройте **Настройки → Устройства и службы → HausmanHub →
   Настроить**.
2. Выберите **Климат**.
3. Выберите **Выбрать комнаты и климатические устройства**.
4. В переходной версии, если HausmanHub ещё не знает систему климата, введите её локальный адрес вида
   `http://192.168.1.10:1880`. Разрешён только буквальный локальный IP без
   пути, параметров, логина и пароля.
5. Выберите комнаты. Затем выберите находящиеся в них устройства. HausmanHub
   показывает формат **«Комната — Устройство»** и не показывает служебные ID.
6. Для каждой выбранной комнаты HausmanHub откроет отдельный короткий экран. Задайте:

   - температуру 18–28 °C, шаг 0,5 °C;
   - влажность 30–70 %, шаг 5 %;
   - мягкую, обычную или быструю стратегию.

   Название контура и режим работы выбираются один раз на первом экране.

7. Проверьте резюме: названия комнат, их отдельные цели, число устройств,
   состояние двигателя и причины, которые требуют внимания.
8. Подтвердите сохранение.
9. Вернитесь в **Климат** и выберите **Задать настройки для дня и ночи**.
   Для каждой комнаты задайте оба набора. Сохранение профилей
   не отправляет команды.
10. Выберите **Настроить автоматическое расписание**, задайте начало дня и ночи.
11. Подтвердите постоянное разрешение. После этого HausmanHub сам выберет нужные
    настройки по местному времени Home Assistant и передаст их подключённой
    системе климата.
12. Для разового изменения откройте **Временно изменить температуру в
    комнате**, выберите комнату и температуру. HausmanHub сразу применит её до
    следующего перехода **«День»/«Ночь»**. Чтобы закончить раньше, используйте
    **Вернуть температуру по расписанию**.

При повторном редактировании HausmanHub заранее отмечает сохранённые комнаты и
устройства. Для каждой комнаты он подставляет сохранённые настройки HausmanHub, а не
временно отличающееся значение двигателя. Редактирование обычных параметров
меняет активный профиль и сохраняет второй без изменений.

Профиль переключается автоматически сразу для всех комнат. Проверка
времени выполняется раз в минуту, но команды отправляются только при настоящей
смене **«День»/«Ночь»**. При включённом расписании ручной выбор закрыт; сами
дневные и ночные значения можно редактировать.

Временная температура хранится отдельно. Например, если для дня сохранено
25 °C, временные 23,5 °C не меняют эти 25 °C. На следующей границе расписания
HausmanHub удалит временное значение и применит сохранённый профиль нового периода.

Публичный API также возвращает точное поле `next_change_at` в местном часовом
поясе Home Assistant. Оно позволяет приложению показать дату и время следующей
смены профиля без собственного пересчёта расписания.

Совместимость формата HausmanHub с уже используемыми приложением типами проверяется
автоматически. Границы этой проверки и ещё не выполненное живое подключение
описаны в [отдельном документе](android-model-compatibility.md).

Полный безопасный цикл создания настройки через HausmanHub API — от списка комнат до
атомарного сохранения — описан в
[руководстве по API настройки](climate-setup-api.md). Сохранение записывает
существующую модель климатического контура, но само не отправляет команды
устройствам. Там же описано чтение текущих профилей и расписания для повторного
открытия редактора без подмены сохранённых значений текущим состоянием двигателя.

## Что означают режимы

- **Наблюдение** — HausmanHub показывает состояние выбранного контура и сравнивает
  цели, но не считает его автоматически активным со стороны HausmanHub.
- **Автоматически** — сохранённые температуру, стратегию и авто-режим можно
  отдельно применить к существующему двигателю. Подтверждение работы появляется
  только при свежих данных, доступных устройствах, готовом праве управления и
  фактически совпадающих значениях.
- **Выключен в HausmanHub** — описание и назначения сохраняются, но HausmanHub не показывает
  контур как активный. Внешний двигатель при этом не выключается.

Если все найденные комнаты уже работают в автоматическом режиме, мастер
предлагает режим **Автоматически** по умолчанию. Иначе безопасным значением
будет **Наблюдение**.

## Что HausmanHub хранит

HausmanHub хранит две связанные части:

- приватный реестр комнат и устройств;
- публичное описание контура и его параметров.

Они проверяются и сохраняются вместе. Если запись одной части не удалась,
предыдущая рабочая настройка восстанавливается.

Устройства имеют понятный тип: кондиционер, радиаторная термоголовка,
увлажнитель, тёплый пол, датчик температуры или датчик влажности. Для устройств,
которыми уже владеет `hausman-climate`, HausmanHub не просит повторно выбирать Home
Assistant entity: приватная связь остаётся у существующего двигателя.

## Статус контура

Перед чтением состояния приложение может запросить локальный маршрут
`GET /api/hausman_hub/v1/capabilities`. Он возвращает контракт
`hausman-hub-capabilities` версии 1: только доступные функции HausmanHub, публичные
пути и версии контрактов. Это описание установленной возможности, а не признак
того, что климат прямо сейчас готов выполнить команду. Фактическая доступность
остаётся в состоянии комнаты и контура.

Основной маршрут приложения `GET /api/hausman_hub/v1/home` возвращает контракт
`hausman-hub-home` версии 12. В одном ответе находятся фактические комнаты и
устройства, а также массив `contours` с состоянием сохранённого климатического
контура. HausmanHub выполняет одно чтение существующего Climate API и строит обе
части из одного результата. Поэтому температура комнаты, доступность устройств
и оценка контура относятся к одному моменту. Отдельный маршрут `/contours`
сохранён для совместимого перехода.

Поле `display_names` содержит единый русский словарь для всех публичных кодов,
включая причины блокировки комнаты и причины состояния контура. Приложение
хранит постоянный код, а человеку показывает название из этого словаря.
Произвольный текст состояния устройства из Climate API HausmanHub никогда
не пересылает: он переводится в один из пяти ограниченных состояний — работает,
ожидает, выключено, недоступно или неизвестно.

У каждой комнаты есть отдельный блок `actual`. Он содержит только прочитанные
температуру, влажность и режим, а `data_status` отличает актуальные данные от
устаревших и отсутствующих. Устаревшее последнее значение можно показать с
предупреждением; при отсутствии комнаты значения равны `null`, а режим —
`unknown`. Плоские поля временно сохранены для совместимого перехода приложения.

В той же комнате `active_target` содержит только текущую цель, прочитанную из
Climate API: температуру, влажность и характер работы. `saved_profiles` отдельно
содержит сохранённые HausmanHub профили **«День»** и **«Ночь»** и выбранный профиль.
Если климатический контур ещё не настроен, профили равны `null`; HausmanHub не копирует
в них активную цель. Так приложение отличает «действует сейчас» от «сохранено
для расписания».

В блоке `control` поле `actions` перечисляет действия, которые поддерживает
выбранное устройство комнаты, а `allowed_actions` — только действия, которые
HausmanHub разрешает выполнить прямо сейчас. Приложение включает кнопку только по
второму списку. Разрешение рассчитывается отдельно для каждой комнаты: свежие
данные и готовность гостиной не открывают управление детской. Общий признак
`climate.commands_enabled` равен `true`, только если разрешённое действие есть
хотя бы у одной комнаты.

`action_availability` содержит отдельное состояние каждого действия из
`actions`: признак разрешения и точные коды причин блокировки. Каждый код имеет
русское объяснение в `display_names.blocked_reasons`. Разрешённое действие не
имеет причин блокировки, а запрещённое не может остаться без объяснения.

`state_revision` — безопасный числовой отпечаток всего публичного состояния,
кроме времени чтения `generated_at`. Если номер не изменился, приложение может
не перестраивать климатический экран. После изменения любого видимого значения
номер станет другим. Это не последовательный счётчик: сравнивать нужно только
равенство старого и нового номера.

Локальный маршрут `GET /api/hausman_hub/v1/contours` возвращает контракт
`hausman-hub-contours` версии 7. Для каждого климата доступны:

- имя, тип, режим и общее состояние;
- комнаты, текущая температура и влажность;
- активные цели и стратегия;
- выбранный профиль и оба набора **«День»/«Ночь»**;
- признак расписания, время начала дня и ночи, следующий профиль и точная
  местная дата его включения;
- временная температура комнаты, правило и точный момент её завершения, а
  также доступность изменения;
- количество назначенных устройств;
- режим существующего двигателя;
- фактическая стратегия существующего двигателя;
- готовность его права управления;
- совпадение целей;
- подтверждение автоматической работы;
- доступность явного применения поддерживаемых настроек;
- ограниченный список понятных причин.

В ответе нет адреса двигателя, `source_id`, `entity_id`, команд нижнего уровня
или приватного содержимого Home Assistant. Маршрут требует аутентификацию Home
Assistant, локальный источник, запрещает CORS и не кэшируется.

## Как HausmanHub 1.5 применяет настройки

Обычный автоматический контур использует отдельный внутренний режим связи
`managed`. Он не меняет смысл прежнего `shadow`: режим **«проверка без команд»**
по-прежнему никогда не отправляет POST. Технический `canary` также остаётся
отдельным старым инструментом одной комнаты.

Перед применением HausmanHub берёт значения выбранного профиля, читает свежее
состояние и строит только необходимые команды. Порядок фиксирован для каждой
комнаты:

1. `set_room_target_strategy` — мягкая, обычная или быстрая стратегия;
2. `set_room_target` — сохранённая температура;
3. `set_room_mode auto` — автоматический режим включается последним.

Команды строятся из сохранённого контура. Планшет и форма Home Assistant не
могут передать тип backend-команды, приватный идентификатор, entity, сервис или
произвольный payload.

До первой отправки HausmanHub резервирует `request_id` и непрозрачный `operation_id`.
Повтор одинакового запроса возвращает прежнюю квитанцию без нового POST.
Неоднозначный сетевой результат также не переотправляется. После всех принятых
команд HausmanHub перечитывает состояние и считает комнату подтверждённой только при
совпадении температуры, стратегии и автоматического режима.

Цель влажности не отправляется: существующий Climate API пока не имеет общей
команды влажности комнаты. Это ограничение присутствует в preview-контракте.

Для Android доступны два локальных маршрута с той же строгой ролью планшета:

- `GET /api/hausman_hub/v1/contours/apply-preview`;
- `POST /api/hausman_hub/v1/contours/apply` с точными полями `request_id`,
  `contour_id=climate` и `confirm=true`.

Для временной температуры используется третий маршрут:

- `POST /api/hausman_hub/v1/contours/temporary-temperature`.

Запрос содержит только открытые идентификаторы контура и комнаты, действие
`set` или `clear`, температуру и явное подтверждение. Для `set` допустимы
18–28 °C с шагом 0,5 °C; у `clear` температуры быть не должно. Произвольную
длительность, внутреннюю команду или приватную привязку передать нельзя.

Перед первой командой HausmanHub сохраняет временное значение. Затем строит план
только для выбранной комнаты. Если результат отправки неизвестен, HausmanHub честно
показывает недоступный результат и не делает автоматический повтор. При смене
периода расписание сначала удаляет все временные значения, сохраняет новый
профиль и только затем один раз применяет его.

Установка или обновление HausmanHub эти маршруты не вызывает и живые настройки не
применяет. Автоматическое расписание использует тот же исполнитель, но только
после отдельного постоянного подтверждения владельца. Сначала HausmanHub сохраняет
новый выбранный профиль, затем выполняет одну попытку применения. Неопределённый
ответ не приводит к автоматической повторной отправке.

## Следующие этапы 1.x

1. Подключить экран Android к контракту климата: состояние, объяснение,
   изменение параметров и временный режим.
2. Добавить непрерывный диспетчер контура, журнал понятных решений и разрешение
   конфликтов между несколькими типами контуров.
3. Добавить следующие контуры на общей основе без изменения принципа работы
   приложения.

## Дополнительные настройки

Старые экраны реестра, bridge/shadow/canary, встроенного предпросмотра 0.6 и
служебного переключателя сохранены для миграции, диагностики и разработки.
Они находятся в **Диагностике и обслуживании** и не нужны для обычной настройки
климата.

Пробное физическое управление одной комнатой остаётся отдельным техническим
инструментом старого rollout. Публикация или установка HausmanHub сама не включает
его и не применяет настройки автоматического контура.
