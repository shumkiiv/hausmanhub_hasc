# Как HausmanHub соблюдает исходные правила дома

Этот документ — короткая карта безопасности. Исходные архитектурные документы
живут в боевом репозитории HausmanHub и читаются здесь только как источник
правил. HausmanHub не копирует оттуда настройки, адреса, названия устройств или
команды.

## Простая картина

HausmanHub сейчас — отдельный фасад просмотра, настройки логических устройств и
ограниченного canary-управления. Прежний `input_boolean` canary остаётся.
Дополнительно HausmanHub начинает владеть климатическими целями и решениями: в 0.6.0
он только рассчитывает потребность одной комнаты. Боевой climate-core временно
остаётся источником показаний и физическим исполнителем с cooldown, authority,
safety и physical feedback.

| Исходное правило | Что это означает для HausmanHub сейчас | Чем это проверяется здесь |
| --- | --- | --- |
| Разделение контуров и ADR | HausmanHub владеет только явно описанной климатической политикой одной комнаты; бытовые сценарии, общее описание дома и другие контуры не затрагиваются. | Граница и этапы описаны в [климатической архитектуре](climate-control-architecture.md), чистый расчёт проверяется синтетическими тестами. |
| Общий контракт дома | HausmanHub может проверять только учебные примеры общего описания, но не меняет его и не получает настоящие данные. | `fixtures/common_contract/` и местная проверка. |
| Реестр автоматизаций | HausmanHub не создаёт и не меняет бытовые сценарии. | Нет файлов сценариев, команд или развёртывания; это проверяет проверка файлов. |
| Граница климата | HausmanHub может хранить цели и показывать своё решение, но 0.6.0 не исполняет его. | Предпросмотр всегда `preview_only` и `commands_enabled=false`; при выключенной связи нет I/O. |
| Роль центра управления домом | HausmanHub не принимает решения за другие части дома. | Два режима наблюдения дополнены только одним явно включаемым helper-canary без устройства. |
| HACS-проверка | Установка через HACS сама не включает управление. | Canary выключен по умолчанию и требует отдельного выбора одного helper. |
| Проверки общего контракта | Проверяются только учебные данные без настоящих названий и адресов. | Три учебные проверки запускаются одной местной командой. |
| Настройки интеграции | Режимы остаются «Только чтение» и «Проверка без изменений»; управление включается отдельным canary-флагом. | Форма отклоняет другой режим, небулевый флаг и любую цель кроме одного `input_boolean`. |
| Диагностика и подсказки | Отчёт содержит только разрешённые общие сведения и статус canary, но не его цель; подсказки ничего не исправляют сами. | Тесты фиксируют allow-list и отсутствие runtime-идентификатора. |
| Сравнение без изменений | Сравнение остаётся недоказанным, пока владельцы контуров не примут отдельное решение. | Учебный пример помечает результат как неразрешённый. |
| Прямое выполнение | Общая власть над устройствами заблокирована; разрешён только helper-canary. | Перед каждой командой проверяются одна запись, флаг, точная цель, домен и доступность helper. |

## Что можно делать без нового решения

- улучшать местные учебные проверки и понятные инструкции;
- проверять безопасность установки, удаления и обновления HausmanHub в пустой
  учебной среде;
- показывать только девять заранее разрешённых общих чисел о доме;
- улучшать уже одобренный canary только внутри одного `input_boolean` и его
  проверенного отката;
- развивать чистый климатический расчёт HausmanHub, нативные наблюдения и
  типизированный план, пока выполнение структурно запрещено;
- исправлять ошибки, которые не расширяют доступ HausmanHub.

## Что нельзя начинать автоматически

- передавать произвольные команды, services или приватные entity ID;
- расширять климатический canary больше одной явно выбранной комнаты без нового
  rollout-решения;
- включать выполнение новых климатических правил без отдельного поэтапного
  решения и физического подтверждения;
- добавлять настоящие названия, адреса, ключи, снимки потоков или команды в
  этот репозиторий.

Текущее отдельное решение для климата описано в
[climate-control-architecture.md](climate-control-architecture.md): сначала
HausmanHub-owned предпросмотр и сравнение наблюдений, затем типизированный план без
исполнения, и только после отдельного разрешения одна проверенная комната с
полным откатом. Для другого физического домена по-прежнему нужно отдельное
решение владельца.
