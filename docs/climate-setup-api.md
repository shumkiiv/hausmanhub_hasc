# API настройки климатического контура HASC

Статус для HASC 1.7.6: готовы строгие форматы доступных комнат, кандидатов
устройств, безопасных предложений назначения, черновика контура и результата
его полной проверки и сохранения. Локальный администратор может получить
варианты формы, создать и проверить черновик, а затем сохранить весь контур.

## Список комнат

Контракт `hausman-hasc-climate-rooms` версии 1 содержит:

- время снимка и состояние данных — актуальны или устарели;
- общий признак, можно ли сейчас выбирать комнаты;
- постоянный ID HASC и понятное имя каждой комнаты;
- признак, что комната уже добавлена в HASC;
- отдельное состояние и признак возможности выбора;
- готовые русские названия всех состояний.

Комнаты всегда сортируются по постоянному ID. Уже настроенная комната сохраняет
имя из HASC, даже если временный источник прислал другое название. Это не даёт
экрану настройки неожиданно менять порядок или подписи.

## Безопасное поведение

- При устаревшем снимке комнаты остаются видимыми, но выбрать их нельзя.
- Если свежий снимок больше не содержит настроенную комнату, HASC сохраняет её
  в списке со статусом **«Комната больше не найдена»**.
- В ответе нет адреса внешнего модуля, его идентификаторов устройств, объектов
  Home Assistant и командных данных.
- Чтение списка ничего не сохраняет и не отправляет команды.

Проверяемый пример находится в
`fixtures/hasc_climate_rooms_v1/rooms.json`, а строгая схема — в
`custom_components/hausman_hub/contracts/v1/climate-rooms.schema.json`.

## Кандидаты устройств

Контракт `hausman-hasc-climate-device-candidates` версии 1 возвращает найденные
и уже настроенные климатические устройства. Для каждого кандидата доступны:

- короткая ссылка вида `candidate_0001`, имя и постоянный ID комнаты;
- доступность, признак уже выполненной настройки и понятное состояние;
- предложенные типы HASC и один рекомендуемый тип;
- ID, комната и тип уже настроенного устройства — только если оно существует.

Типы ограничены шестью понятными вариантами: кондиционер, термоголовка,
увлажнитель, тёплый пол и два вида датчиков. Их постоянные коды сопровождаются
готовыми русскими названиями.

Короткая ссылка намеренно не является постоянным ID устройства. Она действует
только вместе с `snapshot_revision`. Номер меняется, если внутренняя привязка
источника изменилась, даже когда все видимые названия совпадают. API черновика
требует тот же номер и повторно проверяет свежий снимок.
Новое время чтения при полностью одинаковом составе и состоянии кандидатов не
меняет номер само по себе.

Приватный ID источника, объект Home Assistant и команды в ответ не входят.
Уже добавленный кандидат нельзя добавить повторно. Пропавшее настроенное
устройство остаётся видимым, а устаревший снимок блокирует весь выбор.

Проверяемый пример находится в
`fixtures/hasc_climate_device_candidates_v1/candidates.json`, строгая схема —
в `custom_components/hausman_hub/contracts/v1/climate-device-candidates.schema.json`.

## Предложение комнаты

Контракт `hausman-hasc-climate-room-suggestions` версии 1 связывает короткую
ссылку кандидата с одной рекомендованной комнатой. Рекомендация появляется
только из явного `room_id` актуального источника. Название устройства не
участвует в выборе: даже устройство с названием «Увлажнитель гостиной» будет
предложено детской, если именно детская указана в проверенном снимке.

Каждая запись содержит название устройства, предложенную комнату, уверенность
и русскую причину. `confirmation_required` всегда равен `true`: контракт не
назначает устройство автоматически. Принять можно только свежее доступное и
ещё не настроенное устройство.

При устаревших данных все комнаты из рекомендаций удаляются. При пропавшем
устройстве или несовпадении текущей и сохранённой привязки HASC также не
предлагает комнату. Недоступное или неподдерживаемое устройство может сохранить
найденную комнату только как объяснение, но `can_accept` остаётся `false`.

Пример и схема находятся в
`fixtures/hasc_climate_room_suggestions_v1/suggestions.json` и
`custom_components/hausman_hub/contracts/v1/climate-room-suggestions.schema.json`.

## Получить данные формы

`GET /api/hausman_hub/v1/admin/climate-drafts` возвращает контракт
`hausman-hasc-climate-setup-options` версии 1. В одном ответе находятся:

- актуальный `snapshot_revision` для последующей отправки формы;
- комнаты, которые можно выбрать;
- найденные устройства, подходящие типы и безопасное предложение комнаты;
- понятные русские названия режимов, типов устройств, состояний и стратегий;
- общий признак `draft_creation_allowed`.

Короткие `candidate_0001` действуют только внутри состава, обозначенного этим
номером снимка. Приложению не нужно и нельзя вычислять эти ссылки самому.
Проверяемый ответ находится в
`fixtures/hasc_climate_draft_v1/options.json`, строгая схема — в
`custom_components/hausman_hub/contracts/v1/climate-setup-options.schema.json`.

## Создать черновик

`POST /api/hausman_hub/v1/admin/climate-drafts` принимает JSON следующего вида:

```json
{
  "snapshot_revision": 8115739542118920,
  "name": "Климат дома",
  "mode": "automatic",
  "rooms": [
    {
      "room_id": "living",
      "target_temperature": 25.0,
      "target_humidity": 45,
      "strategy": "normal",
      "devices": [
        {
          "candidate_id": "candidate_0002",
          "type": "air_conditioner"
        }
      ]
    }
  ]
}
```

Температура ограничена диапазоном 18–28 °C с шагом 0,5 °C, влажность —
30–70 % с шагом 5 %. Характер работы выбирается из **плавно**, **обычно** или
**быстро**. Одно устройство нельзя повторить или перенести в комнату, которой
нет в свежих данных. Тип должен входить в варианты, реально определённые HASC
для этого устройства.

При успехе HASC возвращает контракт `hausman-hasc-climate-draft` версии 1:
нормализованные комнаты, устройства, цели, русские названия, краткое резюме и
`draft_revision`. Комнаты и устройства сортируются, поэтому одинаковый ввод
создаёт одинаковый номер черновика.

Поля `save_allowed: false` и `validation_required: true` означают, что это
только черновик. Реестр, настройки контура, теневые доказательства готовности
и устройства не изменяются. Строгие примеры запроса и ответа находятся в
`fixtures/hasc_climate_draft_v1`, схемы — в
`climate-draft-request.schema.json` и `climate-draft.schema.json`.

## Проверить черновик

Созданный ответ целиком отправляется на
`POST /api/hausman_hub/v1/admin/climate-drafts/validate`. HASC снова читает
актуальный снимок и проверяет:

- не изменился ли состав или приватная привязка найденных устройств;
- не менялись ли поля и номер самого черновика после его создания;
- есть ли в каждой комнате хотя бы одно управляющее устройство;
- достаточно ли заявленных возможностей для выбранного типа устройства;
- можно ли построить существующую модель климатического контура HASC.

Успешный ответ имеет состояние `ready` и `save_allowed: true`. Это означает
готовность к отдельному сохранению: сама проверка ничего не записывает.
`command_allowed` всегда равен `false`.

Если, например, для комнаты выбран только датчик температуры, HASC возвращает
состояние `blocked`, `save_allowed: false` и объяснение **«В комнате нет
устройства, которое может управлять климатом»**. Неполные возможности
устройства получают отдельное понятное объяснение. Сырые ошибки и приватные ID
наружу не передаются.

Пример готового результата находится в
`fixtures/hasc_climate_draft_v1/validation.json`, строгая схема — в
`custom_components/hausman_hub/contracts/v1/climate-draft-validation.schema.json`.

## Сохранить контур

Тот же неизменённый черновик целиком отправляется на
`POST /api/hausman_hub/v1/admin/climate-drafts/save`. Перед записью HASC под
одной внутренней блокировкой снова читает свежий снимок и выполняет все
проверки предыдущего раздела. Поэтому между проверкой и записью нельзя
незаметно подменить короткую ссылку кандидата другим устройством.

После успешной проверки HASC одной согласованной операцией сохраняет:

- комнаты и приватные привязки выбранных устройств;
- типы устройств и их проверенные возможности;
- название, режим, температуру, влажность и характер работы комнат;
- климатический контур на существующем двигателе `existing_climate_core`.

Если запись реестра, контура или служебного состояния не удалась, HASC
восстанавливает прежнюю рабочую пару. Успешный ответ имеет состояние `saved`,
возвращает номера снимка и черновика, количество комнат и устройств, а также
`commands_sent: false` и `restart_required: false`. В ответе нет приватных ID.
Сохранение меняет только настройку: ни одна команда физическому устройству не
отправляется.

Пример ответа находится в `fixtures/hasc_climate_draft_v1/save.json`, строгая
схема — в
`custom_components/hausman_hub/contracts/v1/climate-draft-save.schema.json`.

## Открыть текущую настройку

`GET /api/hausman_hub/v1/admin/climate-drafts/current` возвращает контракт
`hausman-hasc-climate-current-setup` версии 1. Он предназначен для повторного
открытия сохранённого контура в редакторе и содержит:

- название и режим контура;
- комнаты и назначенные им устройства с понятными типами;
- отдельные сохранённые профили **«День»** и **«Ночь»** каждой комнаты;
- выбранный профиль и отдельно временную температуру, если она действует;
- включение расписания, начало дня и ночи, последний применённый профиль;
- `setup_revision` — номер изменения всей сохранённой настройки;
- `snapshot_revision` — номер актуального состава и привязок устройств.

HASC берёт профили из собственного хранилища. Например, временная температура
23,5 °C показывается отдельно и не заменяет сохранённые 25 °C для дня.

Если контур ещё не создан, ответ имеет состояние `not_configured`. При
устаревших данных, недоступном, пропавшем или изменившемся устройстве сохранённые
значения остаются видимыми со статусом `attention`, но `editing_allowed` равен
`false`. Это не позволяет редактору сохранить старую привязку как актуальную.

Пример находится в `fixtures/hasc_climate_current_setup_v1/current.json`,
строгая схема — в
`custom_components/hausman_hub/contracts/v1/climate-current-setup.schema.json`.

## Доступ и ошибки

- Все операции настройки требуют обычного локального администратора Home Assistant.
- Пользователь планшета не получает административный доступ.
- Запрос с внешнего адреса отклоняется.
- Устаревшие данные или изменившийся `snapshot_revision` дают HTTP 409: форму
  нужно обновить через `GET`.
- Неверные диапазоны, лишние поля, повтор устройств и неподходящий тип дают
  HTTP 400.
- Все ответы HASC имеют `Cache-Control: no-store`.
- Запросы создания, проверки и сохранения черновика ограничены 256 КБ.

Следующий шаг разработки — изменять оба сохранённых профиля комнаты через
строгий запрос, связанный с `setup_revision`.
