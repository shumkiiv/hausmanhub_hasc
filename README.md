# HausmanHub

HausmanHub — система автоматических контуров дома в виде интеграции Home
Assistant. Сейчас развивается первый контур — климат: комнаты связываются с кондиционерами,
термоголовками, увлажнителями, тёплым полом и датчиками. Позднее на той же
основе можно добавить освещение, воду, безопасность и энергию.

**HACS** — это не название проекта, а каталог, через который HausmanHub можно
установить и обновлять в Home Assistant.

Текущая разрабатываемая версия — **1.7.7**. Последний опубликованный релиз —
**1.5.0**.

## Как теперь устроен климат

Обычная настройка выглядит так:

1. Открыть **HausmanHub → Настроить → Климат**.
2. Выбрать **Выбрать комнаты и климатические устройства**.
3. Пока идёт перенос алгоритма, один раз указать локальный адрес уже работающей
   системы климата. В самостоятельной версии этот шаг исчезнет.
4. Отметить комнаты и находящиеся в них кондиционеры, термоголовки,
   увлажнители, тёплый пол и датчики.
5. Для каждой выбранной комнаты отдельно задать комфортную температуру,
   влажность и характер работы.
6. Проверить понятное резюме и сохранить настройки.
7. Выбрать **Задать настройки для дня и ночи** и задать
   два набора комфорта для каждой комнаты.
8. Выбрать **Настроить автоматическое расписание**, указать начало дня и ночи
   и подтвердить постоянное разрешение.
9. После этого HausmanHub сам переключает дневные и ночные настройки по местному
   времени Home Assistant и передаёт их подключённой системе климата.
10. Если до следующего переключения нужно сделать комнату теплее или холоднее,
    выберите **Временно изменить температуру в комнате**. Для досрочного
    возврата выберите **Вернуть температуру по расписанию**.

В обычном мастере показываются только понятные названия комнат, устройств и
действий. Служебные средства находятся в разделе
**Диагностика и обслуживание**.

## Что добавляется в версиях 1.7.0–1.7.7

- Зафиксирован строгий формат списка комнат для будущей настройки климата
  через API HausmanHub. В нём есть постоянный ID, понятное имя, признак уже
  настроенной комнаты и возможность выбрать её сейчас.
- Если данные устарели, выбор закрывается. Если ранее настроенная комната
  исчезла из источника, она остаётся видимой с объяснением, а не удаляется из
  настройки незаметно.
- В ответ не попадают адрес климатического модуля, его внутренние ID и объекты
  Home Assistant.
- Кандидаты устройств теперь имеют постоянные типы HausmanHub и готовые русские
  названия: **кондиционер**, **термоголовка радиатора**, **увлажнитель**,
  **тёплый пол**, **датчик температуры** и **датчик влажности**.
- Для каждого кандидата видно, можно ли его добавить, добавлен ли он раньше,
  доступен ли сейчас и не изменилась ли его привязка. Внутренний ID заменён
  короткой ссылкой, действующей только вместе с номером текущего снимка.
- HausmanHub предлагает комнату только тогда, когда устройство действительно найдено
  в ней по актуальным данным. Название устройства не используется для
  догадок. Рекомендация имеет русское объяснение и требует подтверждения;
  чтение списка само ничего не назначает.
- Локальный администратор может запросить готовые данные формы через
  `GET /api/hausman_hub/v1/admin/climate-drafts`, выбрать значения и отправить
  их на тот же адрес методом `POST`. HausmanHub вернёт понятный черновик, но пока не
  сохранит его и не включит управление.
- Старый номер списка устройств отклоняется: HausmanHub просит обновить форму, а не
  рискует назначить изменившееся устройство.
- Готовый черновик можно отправить на
  `POST /api/hausman_hub/v1/admin/climate-drafts/validate`. HausmanHub повторно
  сверит его со свежими устройствами, проверит наличие управляющего устройства
  в каждой комнате и достаточность возможностей. Ответ объяснит, готов ли
  черновик к сохранению. Сама проверка ничего не сохраняет и не включает.
- Проверенный черновик целиком отправляется на
  `POST /api/hausman_hub/v1/admin/climate-drafts/save`. HausmanHub ещё раз сверяет
  свежесть данных и одной операцией сохраняет комнаты, выбранные устройства и
  их параметры в уже используемую модель климатического контура. При ошибке
  возвращается прежняя настройка. Сохранение не отправляет команды устройствам
  и не требует перезапуска Home Assistant.
- Для повторного открытия редактора локальный администратор запрашивает
  `GET /api/hausman_hub/v1/admin/climate-drafts/current`. HausmanHub возвращает
  сохранённые комнаты, устройства, оба профиля, расписание и временное
  изменение отдельно. Если устройство пропало или данные устарели, настройки
  остаются видны, но изменение блокируется до обновления данных.

## Что добавляется в версиях 1.6.0–1.6.9

- Приложение может запросить
  `GET /api/hausman_hub/v1/capabilities` и узнать, какие функции HausmanHub
  установлены, по каким локальным адресам они доступны и какие версии
  контрактов нужно понимать.
- Ответ не зависит от того, включено ли управление климатом, но требует того же
  локального пользователя планшета. В нём нет адреса климатического модуля,
  объектов Home Assistant или данных комнат.
- `GET /api/hausman_hub/v1/home` возвращает контракт версии 5: в одном снимке
  находятся фактические данные комнат и устройств и состояние сохранённого
  автоматического контура. Они строятся из одного чтения Climate API и не
  расходятся по времени.
- Прежний `GET /api/hausman_hub/v1/contours` пока остаётся доступным, чтобы
  переход на единый ответ был совместимым.
- Контракты `/home` и `/contours` версии 6 используют только постоянные коды
  HausmanHub. В поле `display_names` приложение получает русские названия режимов,
  типов устройств, возможностей, состояний, стратегий, профилей и причин, по
  которым управление сейчас недоступно.
- Сырое состояние устройства, придуманное внешним климатическим модулем, в
  приложение не копируется. Оно безопасно сводится к понятному состоянию:
  **работает**, **ожидает**, **выключено**, **недоступно** или **неизвестно**.
- В контракте `/home` версии 7 каждая комната содержит отдельный блок `actual`:
  текущую температуру, влажность, фактический режим и честную отметку — данные
  актуальны, устарели или пока отсутствуют. При отсутствии данных HausmanHub не
  показывает нули и не придумывает состояние.
- В версии 8 блок `active_target` показывает цель, которая действительно
  действует в климатическом модуле, а `saved_profiles` — отдельно сохранённые
  настройки **«День»** и **«Ночь»**. Поэтому приложение не перепутает текущее
  управление с будущей настройкой расписания.
- В версии 9 дома и версии 7 контуров расписание сообщает следующий профиль и
  точную местную дату его включения со смещением часового пояса. Для временной
  температуры это одновременно точный момент возврата к расписанию.
- В версии 10 дома каждая комната отдельно сообщает `allowed_actions` — только
  действия, которые можно выполнить прямо сейчас. Если управление разрешено в
  гостиной, это не включает кнопки в спальне или детской. Поле `actions`
  остаётся списком возможностей устройства и помогает показать недоступные
  кнопки без ложного разрешения на команду.
- В версии 11 блок `action_availability` отдельно объясняет состояние каждой
  кнопки комнаты: можно ли выполнить действие и какие причины его блокируют.
  Код причины переводится в обычный русский текст через `display_names`,
  поэтому приложение может объяснить человеку, чего именно не хватает.
- В версии 12 поле `state_revision` меняется только вместе с содержимым
  публичного климатического состояния. Одинаковые данные с новым временем
  чтения сохраняют тот же номер, поэтому приложение может не перестраивать
  экран без необходимости. Номера нужно сравнивать только на равенство.
- Для версии 1.6.9 локальная проверка сборки преобразует пример ответа HausmanHub в
  те же простые типы, которые уже использует Android-приложение: комнаты,
  устройства, числа и действия. Она также собирает и проверяет запросы команд,
  поэтому случайное несовместимое изменение API не попадёт в выпуск незаметно.
- Android и существующий климатический модуль остаются внешними системами только
  для чтения со стороны разработки HausmanHub. Текущий Climate API временно служит
  безопасным мостом и эталоном поведения. Алгоритм будет перенесён внутрь HausmanHub;
  после проверки совпадения отдельный климатический модуль для работы не
  понадобится.

## Что делает версия 1.5.0

- Позволяет временно задать температуру одной комнаты, не изменяя сохранённые
  настройки **«День»** и **«Ночь»**.
- Сразу передаёт изменение существующей системе климата только для выбранной
  комнаты и затем проверяет фактическое состояние.
- Автоматически отменяет временную температуру при следующем переходе
  **«День»/«Ночь»**. Её также можно отменить раньше отдельным понятным
  действием.
- Показывает комнату, текущее заданное значение, момент возврата и честный
  результат применения.
- Даёт приложению планшета строгий локальный маршрут
  `POST /api/hausman_hub/v1/contours/temporary-temperature`. Приложение может
  только установить разрешённую температуру или вернуть расписание; оно не
  может передать внутреннюю команду, адрес или длительность.
- Обновляет контракт состояния контуров до v5: для каждой комнаты видны
  активность временной температуры, её значение, правило завершения и
  доступность управления.
- Сохраняет временное намерение до первой физической команды. Если ответ
  потерян, HausmanHub не повторяет возможную команду автоматически.

В версии также остаются все возможности 1.4.0:

- Показывает полностью русский интерфейс по умолчанию, даже если язык браузера
  не совпадает с языком Home Assistant.
- На первой странице оставляет три понятных раздела: **Климат**,
  **Информация о доме** и **Диагностика и обслуживание**.
- Настраивает климат сразу для нескольких выбранных комнат и устройств.
- Хранит понятные параметры комфорта: температуру 18–28 °C с шагом 0,5 °C,
  влажность 30–70 % с шагом 5 % и мягкую, обычную или быструю стратегию.
- Показывает отдельный короткий экран для каждой выбранной комнаты, поэтому у
  спальни, детской и гостиной могут быть разные температура, влажность и
  стратегия.
- Хранит для каждой комнаты два понятных набора комфорта: **«День»** и
  **«Ночь»**. В каждом наборе отдельно задаются температура, влажность и
  характер работы.
- Автоматически выбирает **«День»** или **«Ночь»** по заданному времени и
  применяет выбранные настройки после отдельного постоянного разрешения.
- Не повторяет команды каждую минуту: если нужный набор уже выбран, проверка
  расписания ничего не меняет.
- Позволяет выключить расписание и снова выбирать **«День»** или **«Ночь»**
  вручную.
- Автоматически переносит прежние настройки в новый формат с выключенным
  расписанием, поэтому само обновление текущий климат не меняет.
- При повторном редактировании заранее отмечает сохранённые комнаты и
  устройства и подставляет именно сохранённые настройки HausmanHub, а не случайное
  текущее значение двигателя.
- Перед сохранением показывает сводку параметров по названиям комнат.
- В текущей переходной версии использует существующую систему
  `hausman-climate` для фактического управления. Её проверенные правила будут
  поэтапно перенесены в HausmanHub и сверены до отключения внешней зависимости.
- Показывает «автоматически» только когда существующий двигатель действительно
  находится в авто-режиме, имеет право управления, получает свежие данные,
  видит доступные устройства и использует совпадающие цели.
- Отдаёт приложению планшета отдельный локальный контракт
  `GET /api/hausman_hub/v1/contours` без приватных идентификаторов.
- Сохраняет список устройств и описание контура атомарно: неполная настройка не
  заменяет рабочую.
- По отдельному подтверждению передаёт существующему двигателю три уже
  поддерживаемые настройки: стратегию, температуру комнаты и автоматический
  режим. Автоматический режим всегда применяется последним.
- Перед применением показывает число комнат и команд. Один и тот же запрос
  нельзя выполнить дважды: повтор только возвращает прежний результат.
- После команд заново читает состояние двигателя. HausmanHub различает подтверждённый,
  ожидающий, частичный, отклонённый и недоступный результат и не выдаёт принятый
  HTTP-ответ за подтверждение состояния.
- Добавляет для будущего Android-экрана локальные маршруты предпросмотра и
  применения: `GET /api/hausman_hub/v1/contours/apply-preview` и
  `POST /api/hausman_hub/v1/contours/apply`.
- Контракт состояния показывает активный набор, значения **«День»** и
  **«Ночь»**, расписание и доступность применения настроек.

## Переходная граница версии 1.5.0

В версии 1.5.0 HausmanHub ещё не дублирует проверенный климатический алгоритм. Он
хранит понятные настройки комнат и расписание, а решения, задержки, защиты и
физическое выполнение пока остаются в существующем `hausman-climate`. Это
переходное состояние до проверенного переноса алгоритма внутрь HausmanHub.

Установка или обновление 1.5.0 ничего не применяет автоматически. Расписание
после обновления выключено. Команды по времени появляются только после того,
как владелец сам включит расписание и подтвердит постоянное разрешение.
После включения автоматического режима существующий двигатель может сразу
запустить или остановить выбранные устройства по своим правилам.

Цель влажности пока остаётся желаемым параметром и отображается в состоянии.
Текущий Climate API не предоставляет общей команды влажности комнаты, поэтому
HausmanHub честно не включает её в применение. Следующий этап — подключить готовое
управление и состояние к полноценному экрану Android.

Если климатический контур был создан в HausmanHub 1.0.0, после обновления один раз
откройте **Создать или изменить климатический контур** и снова сохраните его.
Так прежнее подключение «только проверка» будет заменено отдельным режимом
обычной работы контура без нарушения гарантии старого shadow-режима.

Отключение контура в HausmanHub прекращает его представление как активного контура
HausmanHub, но не выключает уже работающий внешний климатический двигатель и не
отправляет команды устройствам.

## Установка через HACS

1. Откройте **HACS** в боковом меню Home Assistant.
2. В меню с тремя точками выберите **Пользовательские репозитории**.
3. Добавьте `https://github.com/shumkiiv/hausmanhub_hacs` с типом
   **Интеграция**.
4. Найдите **HausmanHub**, нажмите **Скачать** и перезапустите Home Assistant.
5. Откройте **Настройки → Устройства и службы → Добавить интеграцию**, найдите
   **HausmanHub** и оставьте режим **Обычный режим — рекомендуется**.

Подробная пошаговая инструкция, обновление и решение частых проблем описаны в
[руководстве по установке HausmanHub через HACS](docs/install-via-hacs.md).

Установка и обновление сами по себе не настраивают живой климат и не отправляют
команды. Право на применение появляется только у сохранённого автоматического
контура и всё равно требует отдельного подтверждения.

## Разделы настроек

- **Климат** — основное место настройки комнат, устройств, дневных и ночных
  значений и расписания.
- **Информация о доме** — девять обезличенных диагностических счётчиков и
  частота их обновления; на климат не влияет.
- **Диагностика и обслуживание** — подключение, перенос старых настроек и
  служебные проверки. Для обычной работы они не нужны.

Подробная схема описана в [руководстве по климатическому
контуру](docs/climate-control-architecture.md), а проверка установки — в
[инструкции для Home Assistant](docs/home-assistant-safe-check.md).

## Безопасность

HausmanHub не хранит в репозитории ключи, пароли, адрес дома, реальные идентификаторы
устройств или снимки настроек. Контур принимает только локальный HTTP(S)-адрес
с буквальным частным IP. Публичные адреса, перенаправления, имена сайтов, пути,
параметры, логины и пароли отклоняются.

Приложение планшета не может передать произвольный сервис Home Assistant,
внутренний идентификатор устройства, адрес или команду нижнего уровня.

## Для разработчиков

Полная локальная проверка:

```shell
python3 tools/check_local_release.py
```

Она использует синтетические данные, не подключается к дому и не отправляет
команды устройствам. Правила разработки описаны в
[CONTRIBUTING.md](CONTRIBUTING.md) и
[инженерных требованиях](docs/engineering-standards.md).

Kimi должен проверить окончательный текущий набор изменений до того, как
изменение будет считаться завершённым или будут выполнены коммит, отправка,
выпуск, развёртывание или публикация.

Если Kimi временно недоступен, другой независимый просмотр может поддержать
работу. Он позволяет продолжать любые изменения внутри границ HausmanHub: код, тесты,
документацию, местные проверки и исправления. Он не разрешает коммит, отправку,
выпуск, развёртывание, публикацию или новые права.

Исключение для изменения только документации действует лишь когда в наборе нет
кода; в смешанном наборе действует финальная проверка Kimi.
