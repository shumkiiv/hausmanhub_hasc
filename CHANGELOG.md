# История версий

## 1.9.6 — в разработке

- Чистый адаптер наблюдений Home Assistant — второй шаг пункта 36 дорожной
  карты. Состояния сущностей через привязки реестра версии 2 превращаются во
  внутреннюю модель наблюдения HausmanHub без внешнего Climate API.
- Комната получает температуру и влажность из наблюдательных привязок
  пассивных датчиков (с запасным чтением текущей температуры из климати-
  ческой сущности), состояние окна из своей привязки, режим и наблюдаемые
  цели из локального контура. Устройство получает активность из состояния и
  физического действия сущности, текущую цель и вентилятор из атрибутов,
  подтверждённые переходы кондиционера — из защищённой памяти перезапуска.
- Дом получает наружную температуру (она же питает правило тепловой
  нагрузки), присутствие и центральное отопление из домашних привязок;
  период дня вычисляется из местного расписания контура. Сезон пока честно
  остаётся неизвестным.
- Отсутствующие, недоступные, устаревшие или нечисловые значения остаются
  честно неизвестными и никогда не превращаются в разрешающие. Адаптер
  чистый: он потребляет абстрактный неизменяемый снимок состояний и не
  импортирует Home Assistant, поэтому проверяется обычными тестами.
- Поведение выполнения не меняется: рантайм, публичные проекции и команды
  идут прежним путём. Работающий климатический модуль и Android-приложение
  не изменялись.

## 1.9.5 — подготовлено в main

- Реестр климата переходит на версию схемы 2: у HausmanHub появляются
  собственные привязки наблюдений Home Assistant. Это первый шаг пункта 36
  дорожной карты — удаление рабочей зависимости от внешнего Climate API.
- Комната может хранить необязательную привязку оконного датчика
  (`binary_sensor.*`). Без привязки состояние окна честно остаётся
  неизвестным и не превращается в разрешающее значение.
- Пассивный датчик температуры или влажности может хранить одну
  наблюдательную привязку (`sensor.*`) строго своей роли. Управляющей
  привязки у пассивного датчика по-прежнему быть не может.
- Новый блок домашнего окружения хранит необязательные привязки наружной
  температуры, присутствия и центрального отопления со строгой проверкой
  доменов сущностей.
- Сохранённый реестр версии 1 мигрирует один раз в версию 2: все новые
  привязки остаются отсутствующими, поэтому миграция не делает старую
  конфигурацию разрешающей. Чтение и запись остаются атомарными.
- Поведение выполнения не меняется: наблюдение и команды по-прежнему идут
  прежним путём. Работающий климатический модуль и Android-приложение не
  изменялись.

## 1.9.4 — подготовлено в main

- Владение устройствами теперь расширяется по одной проверенной комнате.
  Отдельная строгая операция повышает комнату до управления HausmanHub
  только когда все защиты согласны: комната в контуре, режим `canary` или
  `managed`, автоматический контур, свежее наблюдение, готовая комната и
  подтверждённое совпадение плана с наблюдением (сравнение «совпадает»).
- У каждого устройства комнаты должна быть управляющая привязка HausmanHub.
  Повышение атомарно: реестр сохраняется целиком, при ошибке хранилища
  прежний реестр остаётся, а квитанция честно сообщает об отказе. Повторное
  повышение той же комнаты честно отвечает «уже управляется».
- Управляемые комнаты проверяются каждую минуту по той же цепочке защит,
  что и пробная: действие только при расхождении, совпадение пропускается,
  неопределённость закрывает управление без вызовов, выполнение строго по
  закрытому перечню с остановкой при первой ошибке.
- Квитанции повышения редактированы: только постоянный код комнаты, понятный
  статус (повышена, уже управляется, отклонено, ошибка), ограниченные
  причины и счётчики устройств.
- Попутно согласована проверка привязок контура: пробный объём `canary`
  теперь признаётся управлением контура наравне с `managed`. Раньше такая
  комната при запуске честно помечалась ошибкой привязки, хотя пробное
  управление версии 1.9.3 требует именно этот объём. Пассивные датчики
  температуры и влажности законно остаются наблюдаемыми: они не мешают
  повышению своей комнаты и не требуют управляющей привязки.
- Работающий климатический модуль и Android-приложение не изменялись.
  Переводимую комнату оператор убирает из комнат прежнего модуля.

## 1.9.3 — подготовлено в main

- HausmanHub впервые может сам управлять климатом — ровно одной пробной
  комнатой и только при полном согласии всех защит. Пробная комната
  настраивается явно, режим моста должен быть `canary`, контур —
  автоматическим, каждое устройство комнаты — в пробном объёме `canary` с
  управляющей привязкой HausmanHub.
- Проверка идёт каждую минуту и действует только когда внутренний план
  расходится с наблюдаемым состоянием. Совпадение честно пропускается как
  «уже актуально», неопределённость, устаревшие данные, неготовая комната и
  неполный перевод закрывают управление без единого вызова.
- Выполняются только строгие вызовы адаптеров из закрытого перечня, по
  порядку и с остановкой при первой ошибке. Квитанция не содержит объектов,
  полезных нагрузок и ответов: только постоянный код комнаты, понятный
  статус (применено, уже актуально, отклонено, ошибка), ограниченные причины
  и счётчики вызовов.
- Проверенная граница исполнения теперь одна на контур: вызовы Home
  Assistant возможны только в модуле исполнителя пробных вызовов и в
  прежнем canary-переключателе. Автоматическая проверка это фиксирует.
- Работающий климатический модуль и Android-приложение не изменялись.
  Оператор обязан убрать пробную комнату из комнат прежнего модуля, чтобы
  избежать двойного управления.

## 1.9.2 — подготовлено в main

- Внутри HausmanHub появились строгие адаптеры устройств Home Assistant.
  Проверенный внутренний план каждого устройства теперь переводится в точный
  список вызовов: режим кондиционера, целевая температура, ступень
  вентилятора и питание увлажнителя. Других сервисов адаптер назвать не может.
- Адаптер использует только приватную привязку из реестра HausmanHub и
  проверяет возможности устройства. Без управляющей привязки или нужной
  возможности перевод честно останавливается с понятной причиной: нет
  управляющего объекта, нет возможности, действие не поддерживается, режим
  наблюдения или удержания.
- Каждый вызов ограничен: один одобренный сервис, один проверенный объект,
  температура 10–35, влажность 0–100, только известные режимы. Произвольных
  полей и чужих привязок нет. Настройка тихого режима пока честно отмечается
  как непереведённая — у неё нет строгого вызова. Вызов смены режима
  требует заявленной возможности `HVAC_MODE`: адаптер не назовёт вызов,
  которого устройство не объявляло.
- Попутно исправлена политика тёплого пола: в отопительном сезоне она теперь
  даёт понятное действие «задать температуру» вместо внутренней ошибки
  расчёта, из-за которой комната с тёплым полом оставалась без плана.
  Эталонные сценарии это не затрагивает — среди них нет тёплого пола.
- Это только перевод: выполнение появится позже отдельными этапами. Свойство
  `commands_enabled` всегда равно `False`; ни Home Assistant, ни прежний
  климатический модуль вызовов не получают. Работающий модуль и
  Android-приложение не изменялись.

## 1.9.1 — подготовлено в main

- Сравнение решений подтверждено на всех 30 замороженных эталонных сценариях
  работающего модуля. Для каждого сценария построено наблюдаемое состояние
  модуля после его эталонного решения, и вердикт сравнения зафиксирован в
  точной таблице: 19 сценариев совпадают полностью, 8 честно несравнимы по
  понятным причинам (ручной режим, намеренное наблюдение, устаревшие данные,
  непоказываемая модулем активность термоголовки).
- Три сценария с защитой выполнения (повторная задержка, подавление дубля,
  отсутствие полномочий) подтверждённо расходятся по ступени вентилятора:
  замороженный модуль усиливает обдув до среднего, а перенесённый слой
  устойчивости без подтверждённой обратной связи и времени работы этого не
  делает. Расхождение зафиксировано как ожидаемый результат сравнения до
  передачи управления, а не скрыто.
- Комнаты в автоматическом режиме без действия на уровне комнаты (например,
  только корректировка термоголовки) теперь сравниваются по планам устройств,
  а не отмечаются несравнимыми целиком.
- Сравнение по-прежнему не отправляет команды, не содержит приватных привязок
  и всегда имеет `commands_enabled=False`. Работающий климатический модуль и
  Android-приложение не изменялись.

## 1.9.0 — подготовлено в main

- HausmanHub теперь сравнивает собственные решения с наблюдаемым состоянием
  работающего климатического модуля — без единой команды устройствам. Для
  каждой настроенной комнаты и каждого выбранного устройства видно, совпадает
  ли строгий внутренний план с тем, что модуль реально делает.
- У каждой комнаты и устройства одно из трёх понятных состояний: совпадает,
  расходится или пока нельзя сравнить. Причины ограничены фиксированным
  списком: устаревшее наблюдение, отсутствующая политика, ручной режим,
  недоступное устройство, ненаблюдаемые настройки, расхождение активности или
  настроек.
- Ручной режим и намеренное наблюдение честно помечаются как несравнимые, а не
  как расхождение. Уже остановленное устройство не считается требующим
  повторной остановки.
- В сравнении остаются только постоянные коды HausmanHub и одобренные
  значения. Свойство `commands_enabled` всегда равно `False`; запрос читает
  одно наблюдение, ничего не записывает и не отправляет команд ни Home
  Assistant, ни прежнему климатическому модулю.
- Работающий климатический модуль и Android-приложение не изменялись.

## 1.8.9 — подготовлено в main

- Защитные задержки климатических устройств теперь переживают перезапуск Home
  Assistant. Для каждого выбранного кондиционера сохраняются только подтверждённые
  факты переходов: фаза работы, время последнего запуска и остановки и число
  подтверждённых коротких циклов.
- Память защиты хранится в версионированном хранилище Home Assistant отдельно
  для каждой записи конфигурации. В ней остаются только постоянные коды
  HausmanHub; приватных привязок, источников, служб и команд нет.
- При запуске сохранённая память сверяется с текущим реестром: отвязанные и
  перенесённые устройства отбрасываются, а память с временем из будущего после
  смены часов полностью сбрасывается. После перезапуска защита один раз
  консервативно перевзводится по свежим наблюдениям и дальше продолжается
  штатно.
- Отказ хранилища закрывает климатический расчёт понятной ошибкой без
  частичного состояния и без команд. Как и раньше, `commands_enabled=False`,
  и ни Home Assistant, ни прежний климатический модуль команд не получают.
- Работающий климатический модуль и Android-приложение не изменялись.

## 1.8.8 — подготовлено в main

- Весь внутренний климатический конвейер теперь запускается отдельно для
  каждой настроенной комнаты. Ошибка входных данных или одного расчёта даёт
  безопасный результат только этой комнаты и не удаляет готовые политики
  остальных комнат.
- Пропавшее устройство изолируется внутри своей комнаты. При наличии других
  выбранных устройств они продолжают рассчитываться; полностью пустая после
  отказа комната одна получает состояние «расчёт не выполнен».
- Для каждой комнаты добавлены четыре строгих состояния: готово, работает с
  ограничениями, недоступно и расчёт не выполнен. Причины различают
  устаревшие или отсутствующие данные комнаты, пропавшее или недоступное
  устройство и локальную ошибку расчёта.
- Неизменяемый результат принимает только постоянные коды HausmanHub, проверяет
  единое время наблюдения и режим контура, не содержит приватных привязок и
  всегда имеет `commands_enabled=False`.
- Runtime получает один снимок, не записывает историю и не отправляет команды
  ни Home Assistant, ни прежнему климатическому модулю.
- Работающий климатический модуль и Android-приложение не изменялись.

## 1.8.7 — подготовлено в main

- Перенесён полный порядок политик работающего климатического контура:
  отсутствие дома, защитная блокировка, свежесть данных, принудительная
  автоматика, ручной режим и обычная автоматика.
- Ручной режим только наблюдает. В нём нет автоматического плана устройств;
  ручной запрос относится к одной точной комнате и не влияет на соседние.
- Принудительная автоматика отклоняет ручной запрос, но продолжает обычный
  расчёт выбранных устройств.
- Открытое или неизвестное окно, отсутствующая температура и явный запрет
  нагрева или охлаждения формируют безопасную остановку раньше проверки
  свежести и режима комнаты.
- Безопасная остановка относится только к выбранным климатическим устройствам.
  Работающий кондиционер, увлажнитель или тёплый пол отмечается для остановки;
  уже выключенное устройство не получает повторного требования. Термоголовка
  остаётся в наблюдении, чтобы не создавать опасную для отопления уставку.
- Устаревший снимок, подозрительный скачок температуры и отложенное действие
  после смены управления переводят комнату в наблюдение без плана устройств.
- Для всех 30 замороженных ситуаций точно совпадают выбранная политика,
  итоговое действие и упорядоченный список блокирующих причин.
- Рабочий путь строит итог за одно чтение и не изменяет историю наблюдений.
  Результат не содержит приватных привязок и всегда имеет
  `commands_enabled=False`; команды Home Assistant не отправляются.
- Работающий климатический модуль и Android-приложение не изменялись.

## 1.8.6 — подготовлено в main

- Добавлен отдельный чистый слой устойчивой работы кондиционеров и
  увлажнителей. Он получает внутреннее наблюдение и базовый план оборудования,
  но не создаёт намерение, команду или право выполнения.
- Для остановленного кондиционера сохранена точная граница запуска 0,7 °C.
  Работающее устройство удерживает мягкий профиль возле цели и перед
  остановкой переходит на 27 °C со слабым обдувом; жёсткий нижний порог имеет
  приоритет над минимальным временем работы.
- Перенесены минимальные циклы работающего алгоритма: 8 минут работы и 6 минут
  паузы по умолчанию, 5/8 при подтверждённо быстром охлаждении и 10/5 при
  медленном. Подтверждённые короткие циклы увеличивают паузу максимум до
  10 минут. Результат показывает оставшееся время в секундах.
- Ночной профиль поддержания всегда тихий. Точные границы длительности не
  считают защиту активной после истечения последней секунды.
- Подтверждённо слабое охлаждение после выдержки усиливается в два шага:
  сначала средний обдув при прежней уставке, затем целевая температура. Без
  подтверждённой физической обратной связи усиление не выполняется.
- Увлажнитель включает и выключает увлажнение через разнесённые пороги. Для
  цели 45 % это 39/44 %, а при работающем охлаждении или тепловой нагрузке не
  ниже 26 °C — 40/45 %. Для другой цели границы сдвигаются вместе с ней.
  Неподтверждённо закрытое окно требует остановки даже без свежего показания
  влажности.
- План строится только для выбранных устройств. Неизменяемая модель отвергает
  подменённые действия, задержки, пороги, противоречивые входы, разное время
  наблюдения и изменяемые списки.
- Рабочий путь рассчитывает защиту за одно чтение без записи истории, вызовов
  Home Assistant и команд прежнему модулю. Все 30 эталонных ситуаций дают
  однозначный результат.
- Переходный Climate API ещё не передаёт обычному наблюдению время физических
  переключений, подтверждённые короткие циклы и окно. Алгоритм уже принимает
  эти факты, но до собственного адаптера HausmanHub не выдумывает их.
- Работающий климатический модуль и Android-приложение не изменялись.

## 1.8.5 — подготовлено в main

- Добавлен отдельный чистый слой планирования теплового оборудования. Он
  сопоставляет решение комнаты только с настроенными кондиционерами,
  термоголовками и тёплым полом и не создаёт намерение или команду.
- Для кондиционеров перенесены базовые профили работающего алгоритма: мягкий и
  обычный режимы используют безопасную уставку не ниже 26 °C и слабый обдув,

  внешний модуль отдельного полного правила тёплого пола не имел.

  следующего read-only snapshot. Для одной комнаты одновременно допускается
  не более одной pending canary-операции, а вытесненный request ID остаётся в
  bounded fail-closed фильтре текущего runtime и не может создать второй POST.

  auth, preview, atomic save, Android state, shadow action, retry и lookup
  через временный loopback Climate API и измеряет ровно ноль command POST.

- Адрес Climate API может быть только HTTP(S) origin с literal loopback,
