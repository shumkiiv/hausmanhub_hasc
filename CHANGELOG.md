# История версий

## 0.5.2 — 2026-07-17

- Shadow теперь ведёт ограниченное 24-часовое окно доказательств: состояние
  сверяется не чаще одного раза в пять минут, а HASC сохраняет только время,
  публичный ID комнаты, разрешённое имя действия и обезличенные счётчики
  `matched`, `missing`, `moved`, `stale`, `rejected`, `translated`. Приватные
  source/entity, адрес, payload и backend-ответы не сохраняются и не
  возвращаются.
- Evidence привязано хешем к точному климатическому реестру и полностью
  сбрасывается при изменении комнаты, устройства или приватной привязки.
  Повреждённое, будущее или неограниченное сохранённое окно закрывает климат,
  а не даёт canary ложную готовность.
- В локальном мастере климатических устройств можно выбрать комнату и увидеть
  read-only результат `collecting`, `blocked` или `ready`. Тот же результат
  доступен локальному администратору через отдельный версионный маршрут и две
  новые установленные JSON Schema v1; аккаунт планшета доступа к нему не имеет.
- Для `ready` нужны свежая точная привязка, authority текущего climate-core,
  три совпавших наблюдения и успешная shadow-трансляция `set_room_target` и
  `turn_room_off` без аномалий. Настроенный canary до этой готовности остаётся
  закрыт, а его первый исполняемый набор ограничен этими двумя действиями.
- Disposable Home Assistant Core-проверка проверяет новый admin-маршрут в
  `disabled` и `shadow`, его redaction и незавершённый gate после одного
  синтетического наблюдения. Shadow по-прежнему обязан показать ровно ноль
  command POST. Физический canary и изменение live climate-core в релиз не
  входят.

## 0.5.1 — 2026-07-17

- В options HASC появился пошаговый локальный мастер климатического реестра:
  комнаты и устройства добавляются отдельными полями, черновик сначала
  проверяется и сверяется, а атомарная замена выполняется только после второго
  явного подтверждения. Расширенный JSON оставлен как необязательный режим.
- В установленный HACS-пакет входят восемь JSON Schema v1 и совместимые
  синтетические fixtures для Android state/actions, квитанций операций,
  admin import, реестра, preview и readiness.
- Добавлены локальные admin-маршруты preview и readiness. Они возвращают
  только нормализованный статус и счётчики; приватный адрес, source IDs и
  entity IDs в readiness не попадают. В `disabled` preview проверяет реестр
  офлайн и не обращается к Climate API.
- Каждая Android-команда теперь требует `request_id`. HASC создаёт непрозрачный
  `operation_id`, возвращает версионную квитанцию и не повторяет state GET или
  command POST для идентичного retry. Повторное использование request ID с
  другим intent отклоняется.
- Квитанции различают `accepted`, `pending`, `confirmed`, `rejected`,
  `timed_out` и `unknown`. Явный отказ backend фиксируется без раскрытия его
  деталей, а транспортная неопределённость не маскируется под отказ. HTTP
  acceptance canary-команды не считается физическим подтверждением; HASC
  подтверждает только результат точного запланированного устройства из
  следующего read-only snapshot. Для одной комнаты одновременно допускается
  не более одной pending canary-операции, а вытесненный request ID остаётся в
  bounded fail-closed фильтре текущего runtime и не может создать второй POST.
- Disposable Home Assistant Core-проверка теперь проходит настоящий local
  auth, preview, atomic save, Android state, shadow action, retry и lookup
  через временный loopback Climate API и измеряет ровно ноль command POST.
  Физический canary и изменение текущего climate-core в этот релиз не входят.

## 0.5.0 — 2026-07-17

- Добавлен приватный климатический реестр HASC: комнаты, логические устройства,
  роли Home Assistant entities, capabilities, owner и rollout scope. Один
  кондиционер, термоголовка, увлажнитель или тёплый пол может объединять
  несколько приватных entities, не раскрывая их Android-приложению.
- HASC читает и строго проверяет существующий Climate API v1, показывает
  администратору кандидатов и расхождения, но никогда не добавляет устройство
  автоматически. Реестр сохраняется версионно и заменяется целиком только
  через локальный административный API.
- Добавлен локальный Android API состояния и типизированных действий. Обычный
  пользователь планшета видит только стабильные ID HASC; `source_id`,
  `entity_id`, Node-RED, vendor transport и backend payload остаются внутри
  HASC.
- Климатический мост имеет отдельные режимы `disabled`, `shadow` и canary одной
  комнаты. Shadow выполняет все проверки и строит план, но никогда не делает
  POST. Canary требует свежий runtime, authority-ready комнату, точную
  привязку, rollout scope и capability от текущего climate-core.
- Переводчик поддерживает room target/mode/minimum/strategy/off и явные power,
  target temperature, target humidity, HVAC и fan команды для кондиционеров,
  термоголовок, увлажнителей и тёплого пола. Произвольный proxy, service,
  entity, URL или backend command от клиента не принимается.
- Адрес Climate API может быть только HTTP(S) origin с literal loopback,
  RFC1918 или ULA IP. Redirect, DNS-имя, публичный адрес, путь, credentials,
  query, большой ответ и повреждённый контракт закрываются.
- Это подготовленный backend и локальный rollout-контур. Живой дом и Android
  ещё не переключены; сначала требуется shadow, затем отдельный canary одной
  комнаты с немедленным откатом в `disabled`.

## 0.4.0 — 2026-07-17

- В настройках появился первый явно включаемый управляющий canary. Он создаёт
  один переключатель HASC и может вызвать только стандартные команды
  включения и выключения для одного выбранного помощника `input_boolean`.
- По умолчанию canary выключен. Выключение удаляет его цель из настроек и
  закрывает управляющий переключатель. Свет, климат, замки, шторы, скрипты,
  сцены и другие домены HASC отклоняет.
- Перед каждой командой HASC заново проверяет единственную сохранённую запись,
  включённый canary и точное совпадение цели. Пропавший или недоступный
  помощник не получает команду.
- Диагностика показывает только факт включения и постоянную область canary,
  но никогда не показывает идентификатор выбранного помощника.
- Привязанная к `input_boolean` автоматизация может запустить другие действия.
  Для первого безопасного теста помощник не следует связывать с устройствами.
  Общего управления устройствами, proxy и прямого протокола устройств в этой
  версии ещё нет.

## 0.3.18 — 2026-07-17

- В безопасном файле диагностики раздел `entry_summary` теперь показывает
  только три уже проверенные настройки самой HASC: режим, включена ли
  дополнительная локальная страница и выбранные 5, 15 или 30 минут обновления.
- HASC не копирует в диагностику сохранённые настройки целиком. Для старой
  записи без новых полей видны безопасные значения по умолчанию: страница
  включена и обновление раз в 5 минут.
- Если HASC выключена, не запущена, повреждена или имеет две записи,
  диагностика по-прежнему содержит только сообщение «недоступно» и не читает
  дом. Новых данных, строк, адресов, устройств, служб, команд или управления
  домом нет.

## 0.3.17 — 2026-07-17

- В параметрах HASC можно выбрать, как часто обновлять те же девять общих
  чисел: раз в 5, 15 или 30 минут. По умолчанию остаются прежние 5 минут.
- Выбор может только оставить прежнюю частоту или уменьшить число местных
  чтений. Более частого варианта нет.
- Настройка не добавляет новые числа, данные, строки, адреса, устройства,
  службы, команды или управление домом. Дополнительная локальная страница
  по-прежнему отвечает сразу на каждый разрешённый запрос и не зависит от
  таймера обычных девяти строк.

## 0.3.16 — 2026-07-16

- В параметрах HASC появился простой выбор для дополнительной локальной
  страницы с девятью общими числами. Её можно выключить, не удаляя HASC.
- После выключения девять обычных строк HASC и безопасный файл диагностики
  остаются как были. Если старый адрес страницы всё ещё открыт, он отвечает
  только «сводка недоступна» и не считает числа для этого запроса. Девять
  строк продолжают обновляться из тех же уже разрешённых общих чисел.
- Включение возвращает только тот же уже разрешённый адрес и те же девять
  чисел. Нового адреса, данных, устройства, службы или управления домом нет.
- Настройка принимает только обычное «включено» или «выключено»; текстовые и
  другие похожие значения безопасно отклоняются.

## 0.3.15 — 2026-07-16

- У девяти уже разрешённых строк появились постоянные обычные значки, чтобы
  их было легче отличать в интерфейсе. Новых чисел, данных или управления это
  не добавляет.
- Дополнительная локальная страница HASC теперь отклоняет все варианты пустого
  сетевого адреса до чтения девяти общих чисел. Это не меняет доступ для
  настоящего домашнего адреса и не добавляет управление домом.
- Страница принимает только обычные адреса домашней сети и отклоняет
  служебные, тестовые и другие специальные адреса до чтения сводки.
  Это не добавляет новый доступ или управление домом.
- Если безопасная сводка неожиданно не может быть прочитана, дополнительная
  страница показывает только «сводка недоступна» без части чисел или причины.
- У этой страницы закреплён один точный адрес без запасных адресов. Это не
  добавляет новый способ доступа или управление домом.
- Проверка также подтверждает, что два редких способа обращения к странице
  сразу закрыты и не могут прочитать или показать девять чисел.
- Проверка подтверждает, что вариант адреса с лишней косой чертой или
  дописанными данными не создаёт вторую страницу и не может прочитать или
  показать девять чисел.

## 0.2.0 — 2026-07-14

Первая опубликованная безопасная версия HASC.

- Два режима: «Только чтение» и «Проверка без изменений».
- Настройки, безопасное удаление и обновление проверены в пустой учебной
  среде Home Assistant.
- Диагностика показывает только девять общих чисел о доме без названий,
  показаний, адресов или ключей.
- Локальная страница с этими же девятью числами доступна только специальному
  пользователю Home Assistant для просмотра и не принимает команды.
